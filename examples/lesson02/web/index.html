WAR file (Web Application folder) Structure

Here is a review of the structure of a deployed Web Application as a WAR file.

A WAR file is a zipped file with the files and folders required for your Web Application to work.

  • webApp.war
    1

    • WEB-INF
      • classes
        • Java Packages as folders
          • Java Class files
          • Java Servlets
        The classes folder contains the compiled Java classes for the application.
      • lib
        • java jar files
        The lib folder contains Java jar libraries. Classes inside jar files will be available in the application later in this course.
      • other private folders
        • private files
        You can have more folders or files inside the web.xml folder. Files inside this folder are not available directly from a web browser.
        Trying to access any resource inside this folder from a web browser (using http://localhost/WEB-INF/url), will trigger a HTTP 404 error response.
        The files inside WEB-INF are available for Java classes usually for configuring the application.
      • web.xml
        The web.xml file is the main configuration file for the web Application.
        This file is not required if using annotations to configure web components, but you are likely to find it useful since some configuration options are only setup in this file.
      • weblogic.xml
        The weblogic.xml file is a server-specific configuration file.
        As in with Oracle's WebLogic Server configuration, you can find server-specific configurations for other servers.
        These files are likely named differently, allowing you to have multiple configurations depending on the target server.
      • other config files
        The WEB-INF folder usually stores more configuration files for either segments of the application or other frameworks
        Some example files:
        • beans.xml : Configures Contexts and Dependency Injection
        • faces-config.xml : for Java Server Faces Configuration
    The WEB-INF folder is a private configuration folder that cannot be accessed by browsers using a URL address.
    The contents of this folder are used by the server and Java classes for configuration purposes.
    They can also be used for private resources, like localized string files and secured web pages.

    • META-INF
      • MANIFEST.MF
    The META-INF folder is another private folder that contains the JAR manifest file and metadata information for the bundle.
    Some frameworks may use this folder to store configurations.

    • images
      • image1.gif
      • image2.jpg
    • styles
      • style.css
      • background.jpg
    • adminPages
      • index.html
      • admin.jsp
      • logo.jpg
    • resources
      • scripts.js
    Files inside a WAR file are accessible by a web browser.
    The exceptions to this are the META-INF and WEB-INF folders.
    To keep a web application organized, you can use folders to store resources just as you would normally do in a web site.
    It is a good idea to keep application resources organized.

    • homeStyles.css
    • topBg.jpg
    • index.html
    • detailView.jsp
    • item.jpg
    Files inside the WAR file are accessible by a web browser.
    You can have all kind of files inside a WAR file and they will be usually sent to the browser.
    The exception to this rule are JSP files, which are dynamic files that may contain Java code that will be executed in the server.

What about servlets?

Servlets are Java classes that are mapped to a resource that most of the time is not a "direct access file".
For example: A HelloServlet class that is mapped to the /Hello resource. This means that when the browser goes to the URL: "http://www.MySite.com/MyApplication/Hello", the HelloServlet class is executed. In the WAR file, the "Hello" file might not even exist.
If there is not a mapping Servlet for a resource, then the Server tries to serve the file with the resource name instead (alias mapping).

Web Application Context Path

A Java web server can have multiple applications running inside. Each application is mapped to a context path.

The context path is the part of the URL between the root of an application and a server address. See the following examples:

  • http://www.myServer.com/shoppingApplication/index.html
  • http://www.myServer.com/shoppingApplication/images/background.jpg
  • http://www.myServer.com/shoppingApplication/style/normal/styles.css

Context Paths are configured in the server. Any application may specify the Context Path as part of its configuration attributes.
It is possible to have a / Context Path, this is commonly used when there is only one application running in a server.

  • http://www.myServer.com/index.html
  • http://www.myServer.com/images/background.jpg
  • http://www.myServer.com/style/normal/styles.css

Remember to be organized with your Context Paths and applications that are deployed to your server, since you may end up having scenarios like the following:

  • http://www.myServer.com/index.html
  • http://www.myServer.com/adminApplication/index.html
  • http://www.myServer.com/searchApplication/index.html

You can obtain the Context Path for an application from a Servlet using 

getServletContext().getContextPath()
